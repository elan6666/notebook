# UE 客户端（系统向）零基础学习计划（24 周｜1h/天）
## 第 3/3 部分：Week 17–24（性能/内存/GC/崩溃定位 + 打包发布 + Lyra 架构阅读 + 作品集收口）

> 每天 60min：10min 回顾 + 25min 学习/阅读 + 25min 实操/提交  
> 本部分核心交付物：  
> 1）**性能基线报告（Stat + Insights）** 2）**内存/泄漏定位报告（Memory Insights + LLM）**  
> 3）**GC 卡顿治理方案（增量 GC + 配置）** 4）**崩溃定位与上报链路（Crash Reporter + asserts）**  
> 5）**可发布包（Packaging/Cooking）** 6）**Lyra 架构拆解 + 你的项目对齐改造点**  

---

## 使用说明（优化版）
- **本部分重点**：从“能写功能”升级到“能定位问题、能交付、能讲工程故事”
- **执行原则**：所有优化、调参、修复都必须留下证据：截图、日志、trace、报告、commit
- **完成标准**：每周至少有 1 份报告型产物，而不只是口头理解

## 第三部分的学习策略
1. **Week 17–20**：围绕性能、内存、GC 建立“测量 → 定位 → 改动 → 复测”的闭环
2. **Week 21–22**：补齐稳定性与交付链路（崩溃、打包、版本说明）
3. **Week 23–24**：对齐 Lyra，并把所有成果打包成作品集与面试材料

## 最终作品集应该包含
- 一个可运行 Demo
- 一份性能报告
- 一份内存/GC 报告
- 一套冒烟测试说明
- 一份打包与运行说明
- 三个可在面试中讲清楚的工程案例

## Week 17｜性能分析入门：先定位瓶颈再优化（Stat/控制台命令）
### 本周目标
- 能回答并验证：瓶颈在 **Game Thread / Render Thread / GPU** 哪一侧
- 建立“性能基线”与最小优化闭环（采样→定位→改动→复测）

### Day 1（性能/分析总览：建立路线图）
- 学：性能与分析整体思路（先测量再行动）
- 做：读完写一页 `docs/perf-workflow.md`：你将如何做性能定位（步骤列表）
- 链接：[性能与概要分析概述（官方，4.27）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/performance-and-profiling-overview?application_version=4.27)

### Day 2（Stat unit：判断瓶颈归属）
- 学：`stat unit` 的含义（Frame/Game/Draw/GPU）
- 做：在 PIE 或 Standalone 运行，记录 3 次采样截图/数值到 `docs/perf-baseline.md`
- 链接：[如何改善游戏线程 CPU 性能（官方博客）](https://www.unrealengine.com/zh-CN/blog/how-to-improve-game-thread-cpu-performance)

### Day 3（Stat Commands：常用 Stat 工具箱）
- 学：常见 stat 命令、采集文件 `stat startfile/stopfile`
- 做：跑 `stat unit`、`stat game`、`stat slate`（若有 UI）、记录结论
- 链接：[Stat 命令（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/stat-commands-in-unreal-engine)

### Day 4（控制台命令参考：知道去哪查）
- 学：控制台命令索引（遇到瓶颈快速找“观测命令”）
- 做：把你项目最常用的 10 个命令整理成 `docs/console-cheatsheet.md`
- 链接：[控制台命令参考（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/unreal-engine-console-commands-reference)

### Day 5（最小优化闭环：只做 1 个改动）
- 学：优化的“最小可验证单位”（一次只改一处，防止混淆）
- 做：选一个明确热点（比如 UI 刷新过频/事件重复）做最小优化并复测 `stat unit`
- 链接：[UMG 最佳实践（官方文档，复查 UI 刷新原则）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/umg-best-practices-in-unreal-engine)

### Day 6（复盘：写性能基线报告 v1）
- 学：用工程语言写结论（现象→证据→动作→结果）
- 做：输出 `docs/perf-baseline-v1.md`（包含：测试场景、stat 数据、改动与前后对比）
- 链接：[Stat 命令（官方，回看采样与文件输出）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/stat-commands-in-unreal-engine)

### Day 7（轻量：把采样场景固定成“Benchmark Map”）
- 学：稳定复测的重要性（相同地图/相同操作）
- 做：建立 `Map_Benchmark`（固定镜头/固定 UI/固定操作脚本都可选），写复测步骤
- 链接：[性能与概要分析概述（官方，复查基线思路）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/performance-and-profiling-overview?application_version=4.27)

#### 每周验收点
- 你能用数据证明：瓶颈主要在 Game/Draw/GPU 哪一侧
- 有 `perf-baseline-v1.md`（含 1 次优化前后对比）

#### 推荐练习题
- 把 UI 更新从“频繁绑定/轮询”改成“事件驱动”并用 Stat 证明收益  
- 做一个“高负载按钮”：一键生成 200 条列表项，观察 `stat slate` 变化

## Week 18｜Unreal Insights：Trace 采集 → 分析 → 写结论
### 本周目标
- 能独立完成一次 Insights Trace（采集、打开、定位热点）
- 学会 Trace 基础概念（Trace Server / .utrace / 控制台/启动参数）

### Day 1（Trace Quick Start：跑通采集链路）
- 学：如何启动 Unreal Insights、如何开始/停止 Trace
- 做：按 Quick Start 完成一次 trace，并保存会话文件；写 `docs/insights-quickstart.md`
- 链接：[Trace 快速入门（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/trace-quick-start-guide-in-unreal-engine)

### Day 2（Unreal Insights 总览：界面与核心视图）
- 学：线程时间线、事件、过滤与定位
- 做：打开昨日 trace：定位 1 个峰值区间，截图+标注写进 `docs/insights-findings.md`
- 链接：[Unreal Insights（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/unreal-insights-in-unreal-engine)

### Day 3（Trace in UE5：框架组成与文件格式）
- 学：TraceLog/TraceAnalysis、Trace Server、.utrace/.ucache
- 做：写一页 `docs/trace-concepts.md`（10 条要点：文件、服务器、会话）
- 链接：[UE5 Trace（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/trace-in-unreal-engine-5)

### Day 4（Trace Control Tab：运行时控制采集）
- 学：如何在 Insights 里控制正在运行会话的 tracing
- 做：在 PIE/Standalone 运行中，尝试通过 Trace Control 控制采集；记录步骤
- 链接：[使用 Trace Control 标签页（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/using-the-trace-control-tab-in-unreal-insights-for-unreal-engine)

### Day 5（Developer Guide：插桩思维（系统岗加分））
- 学：如何为自己的系统/插件写 trace 事件（理解即可，不必深挖）
- 做：读完写 `docs/tracing-plan.md`：你想在 UI 系统里插哪 3 个事件点（例如 Push/Pop、列表刷新、事件广播）
- 链接：[Trace 开发者指南（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/developer-guide-to-tracing-in-unreal-engine)

### Day 6（（可选）Trace Bookmark：给关键逻辑打书签）
- 学：用书签标记关键路径，便于在 Insights 里对齐峰值
- 做：在关键逻辑处加 Bookmark（Blueprint 或 C++ 任一），复采集一次 trace，对比更易定位
- 链接：[Trace Bookmark（Blueprint API，英文）](https://dev.epicgames.com/documentation/en-us/unreal-engine/BlueprintAPI/Perf/InsightsTrace/TraceBookmark)

### Day 7（复盘：输出 Insights 性能报告 v1）
- 学：把“看图”转化为“结论与行动”
- 做：输出 `docs/insights-report-v1.md`（包含：trace 条件、峰值区间、最可疑模块、下一步改动点）
- 链接：[Unreal Insights（官方，回看分析目标）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/unreal-insights-in-unreal-engine)

#### 每周验收点
- 你能独立采集并分析一份 trace
- 你能指出 1 个明确热点（函数/系统/模块）并给出改动建议

#### 推荐练习题
- 做 2 个场景 trace：`Map_Benchmark` vs “高负载 UI”（列表 200 项），对比热点差异  
- 给 UIManager 的 Push/Pop 打书签，验证峰值是否与页面切换一致

## Week 19｜内存定位：Memory Insights + LLM（泄漏/峰值/长期分配）
### 本周目标
- 能用 Memory Insights 找“什么时候分配爆了、哪些 tag 占用大”
- 学会 LLM（Low Level Memory Tracker）基本用法与标签思维

### Day 1（Memory Insights：工具与查询能力）
- 学：Memory Insights 能回答什么（泄漏、峰值、时间窗分配）
- 做：跑一次内存 trace（短即可），打开 Memory Insights，截图主视图到 `docs/memory-insights-notes.md`
- 链接：[Memory Insights（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/memory-insights-in-unreal-engine)

### Day 2（Call Stack Symbol Tracing：符号与调用栈）
- 学：内存调用栈符号追踪需要调试信息文件匹配
- 做：读完写 `docs/symbol-tracing.md`：你项目如何保证符号可用（Debug/Development 的策略）
- 链接：[Memory Insights（官方，包含符号追踪说明）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/memory-insights-in-unreal-engine)

### Day 3（LLM：低级内存跟踪器入门）
- 学：LLM 标记（Tag）与内存归因思想
- 做：启用/使用 LLM（按文档），输出一次 LLM 统计并记录到 `docs/llm-baseline.md`
- 链接：[使用 LLM 跟踪器（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/using-the-low-level-memory-tracker-in-unreal-engine)

### Day 4（LLM + Memory Insights 联动：从 Tag 到调用栈）
- 学：用 LLM 先看“哪类占用大”，再用 Memory Insights 查“谁分配的”
- 做：选一个占用较大的 Tag，追到一次分配调用栈（能看到就截图写结论）
- 链接：
  - [使用 LLM 跟踪器（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/using-the-low-level-memory-tracker-in-unreal-engine)
  - [Memory Insights（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/memory-insights-in-unreal-engine)

### Day 5（移动端补充认知：Android 上的 Memory Insights）
- 学：移动端低内存终止（LMK）与内存工具的关系（了解即可）
- 做：读完写 `docs/mobile-memory.md`：如果面试问移动端内存问题，你怎么说
- 链接：[Android App Quality：Unreal Memory Insights（Google）](https://developer.android.com/topic/performance/vitals/lmk?hl=zh-cn)

### Day 6（复盘：写内存报告 v1）
- 学：用“时间窗 + Tag + 调用栈”讲清一个内存问题
- 做：输出 `docs/memory-report-v1.md`（峰值时刻、Top Tags、可疑分配点、行动项）
- 链接：[Memory Insights（官方，回看查询功能）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/memory-insights-in-unreal-engine)

### Day 7（轻量：制造一次小泄漏并抓到它）
- 学：验证工具链有效性（面试加分：能演示）
- 做：制造“长期分配未释放”（小范围即可），重新采集并证明你能看到异常
- 链接：[Memory Insights（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/memory-insights-in-unreal-engine)

#### 每周验收点
- 有 `memory-report-v1.md`（包含：峰值、Tag、调用栈/证据）
- 你能解释：LLM 与 Memory Insights 各自回答什么问题，如何联动

#### 推荐练习题
- 让“高负载 UI”（列表 200 项）产生明显内存变化，并用 Tag/时间窗解释  
- 把 UI 资源缓存改为弱引用/可回收策略，复测峰值变化

## Week 20｜GC 与卡顿治理：增量 GC + 配置 + 观测
### 本周目标
- 理解 GC 卡顿来源（可达性分析/清扫等）与 UE 的增量 GC 思路
- 会在 Project Settings 里做合理配置，并能用数据验证“卡顿改善”

### Day 1（增量 GC：概念与收益）
- 学：UE 增量垃圾回收做什么、适合什么场景
- 做：读完写 `docs/incremental-gc-notes.md`：你项目是否适合启用、潜在风险点
- 链接：[增量垃圾回收（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/incremental-garbage-collection-in-unreal-engine)

### Day 2（GC Settings：项目设置中的关键开关）
- 学：并行 GC、增量 BeginDestroy 等配置项含义
- 做：打开 Project Settings，逐项对照文档写 `docs/gc-settings.md`（你会怎么配 + 为什么）
- 链接：[垃圾回收设置（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/garbage-collection-settings-in-the-unreal-engine-project-settings)

### Day 3（Stat 观测：把 GC 行为量化）
- 学：用 stat/日志观测 GC（至少做到“看见它发生”）
- 做：在 Benchmark 场景运行，记录 GC 相关统计/日志（写入 `docs/gc-observations.md`）
- 链接：[Stat 命令（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/stat-commands-in-unreal-engine)

### Day 4（用 Insights 看 GC 卡顿：对齐时间线）
- 学：把 GC 事件与帧时间峰值对齐（数据证据）
- 做：采集一次 trace，在峰值处标注“可能的 GC 区间”，截图写结论
- 链接：[Unreal Insights（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/unreal-insights-in-unreal-engine)

### Day 5（改动：启用/调整增量 GC 并复测）
- 学：一次只改一组 GC 相关设置，复测验证影响
- 做：按 Day2 的计划调整 1–2 个开关，复测 `stat unit` + trace（选其一）
- 链接：
  - [增量垃圾回收（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/incremental-garbage-collection-in-unreal-engine)
  - [垃圾回收设置（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/garbage-collection-settings-in-the-unreal-engine-project-settings)

### Day 6（断言与预防：用 check/ensure 捕捉“先用后初始化”等 GC 相关坑）
- 学：check/verify/ensure 的使用边界（开发期快速暴露问题）
- 做：在 UI/Subsystem 入口加 3 处合理断言（例如 ensure(Subsys!=nullptr)），写原因
- 链接：[断言（Asserts）（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/asserts-in-unreal-engine)

### Day 7（复盘：GC 治理报告 v1）
- 学：把“设置/证据/效果”写清楚
- 做：输出 `docs/gc-tuning-report-v1.md`（配置、前后对比、注意事项、回滚策略）
- 链接：[增量垃圾回收（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/incremental-garbage-collection-in-unreal-engine)

#### 每周验收点
- 有 `gc-tuning-report-v1.md`（含配置与复测证据）
- 你能解释：为什么你的项目要/不要启用增量 GC

#### 推荐练习题
- 制造一次对象大量创建/销毁场景（例如反复打开/关闭复杂 UI），观察 GC 峰值  
- 用 ensure 把“错误用法”提前暴露（比如 UIManager 未初始化就调用）

## Week 21｜崩溃定位与上报链路：Crash Reporter + 调试/断言 + 蓝图调试
### 本周目标
- 能把崩溃从“现象”定位到“可复现步骤 + 调用栈 + 证据”
- 知道如何启用/配置 Crash Reporter，并能写出可提交的 Bug 报告

### Day 1（Crash Reporting：Crash Reporter 与数据内容）
- 学：Crash Reporter 客户端、崩溃报告包含什么、如何配置
- 做：读完写 `docs/crash-reporting-notes.md`（你项目如何启用、报告里你关心什么字段）
- 链接：[崩溃报告（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/crash-reporting-in-unreal-engine)

### Day 2（Packaging 设置：Include Crash Reporter）
- 学：打包时是否包含 Crash Reporter（发布策略）
- 做：在项目设置里找到相关选项并记录：开发包/发布包分别怎么配
- 链接：[项目设置：Project 分段（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/project-section-of-the-unreal-engine-project-settings)

### Day 3（断言体系：check/verify/ensure 的取舍）
- 学：开发期用 check/ensure 提前抓根因，减少“随机崩溃”
- 做：把你项目里最容易出错的 3 个入口加上合适断言，并写理由
- 链接：[断言（Asserts）（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/asserts-in-unreal-engine)

### Day 4（日志分类：自定义 Log Category）
- 学：让日志“可过滤、可归因、可复盘”
- 做：为你的 UI 插件新增 1 个自定义 LogCategory；把关键日志切到该类别
- 链接：[自定义日志类别（官方社区 Snippet）](https://dev.epicgames.com/community/snippets/Qg/defining-custom-log-categories-in-c?locale=zh-cn&ref=blog.rushdownstudio.com)

### Day 5（蓝图调试：断点/单步/变量）
- 学：蓝图调试器（定位 UI/流程问题的必备）
- 做：在一个 Widget Blueprint 上打断点，复现一次逻辑，记录调试截图
- 链接：[蓝图调试器（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/blueprint-debugger-in-unreal-engine)

### Day 6（如何提交高质量 Bug：可复现 + 证据）
- 学：Bug 报告结构与 Epic 的建议（可迁移到任何团队）
- 做：写一个 `docs/bug-report-template.md` 并用你项目里的一个问题填一次
- 链接：[报告错误（官方）](https://www.unrealengine.com/zh-CN/support/report-a-bug)

### Day 7（演练：故意制造一次崩溃 → 报告捕获 → 根因与修复）
- 学：把链路跑通（面试展示点）
- 做：制造一个可控崩溃（空指针/ensure 触发），拿到报告/调用栈/日志，修复并写复盘
- 链接：[崩溃报告（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/crash-reporting-in-unreal-engine)

#### 每周验收点
- 有 `bug-report-template.md` + 1 个真实填充样例
- 你能展示：崩溃如何被捕获、如何定位、如何修复（含证据）
