# UE 客户端（系统向）零基础学习计划（24 周｜1h/天）
## 第 2/3 部分：Week 9–16（UMG/Slate + 插件/Subsystem + UBT + 自动化测试）

> 每天 60min：10min 回顾 + 25min 学习/阅读 + 25min 编码/提交  
> 本部分核心交付物：**可复用 UI 系统（UMG） → 理解并能写最小 Slate → UI 系统插件化 → Subsystem 承载 → 掌握 Build.cs/Target → 写并运行自动化测试**

---

## 使用说明（优化版）
- **本部分重点**：从“会做 UI”过渡到“会设计 UI 系统”
- **执行原则**：UI 相关任务优先做最小可用版本，不追求美术效果，重点在结构、生命周期、依赖、可测试性
- **完成标准**：每周都要产出一个“可演示系统”，而不是只记概念

## 第二部分的学习策略
1. **Week 9–10**：先把 UMG 做顺，建立页面、数据、事件的最小闭环
2. **Week 11–14**：把 UI 从“散代码/散蓝图”提升到“系统化 + 插件化 + Subsystem”
3. **Week 15–16**：把工程能力补齐到构建系统与自动化测试

## 每周固定检查项
- 页面/系统是否只有一个入口？
- 生命周期是否清晰？
- 依赖是否收口？
- 是否能写出最小测试或最小复盘文档？

## Week 9｜UMG 基础：Widget Blueprint + 显示/隐藏 + 绑定
### 本周目标
- 能独立完成：**创建 Widget Blueprint → 显示到屏幕 → 响应按钮 → 数据更新**
- 建立 UI 基础目录规范与命名规范（为后续“系统化 UI”做准备）

### Day 1（UMG 快速入门：HUD + 菜单流程）
- 学：UMG 的基本工作流（HUD、前端菜单）
- 做：按快速入门跑通一个 HUD + 简单菜单
- 链接：[UMG UI 设计器快速入门（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/umg-ui-designer-quick-start-guide-in-unreal-engine)
- 产出：`WBP_HUD`、`WBP_MainMenu`（至少能显示）

### Day 2（控件蓝图：Widget Blueprint 的界面与创建方式）
- 学：Widget Blueprint 结构、Designer/Graph、常见控件
- 做：新建 2 个 Widget Blueprint（按钮/文本），加到 Viewport
- 链接：[控件蓝图（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/widget-blueprints-in-umg-for-unreal-engine)
- 产出：`WBP_ButtonDemo`（按钮点击能触发 Print/日志）

### Day 3（属性绑定：绑定变量/函数到 UI）
- 学：Property Binding 的机制与代价（“方便但可能更耗”）
- 做：给 HUD 绑定一个数值（分数/金币），变化后 UI 自动更新
- 链接：[UMG 属性绑定（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/property-binding-for-umg-in-unreal-engine)
- 产出：`WBP_HUD` 显示 `Score`（绑定生效）

### Day 4（创建控件模板：复用按钮/组件）
- 学：Widget Template 思路：把可复用 UI 做成“模板控件”
- 做：做一个 `WBP_CustomButton`（图标+文字+点击事件），嵌入 HUD
- 链接：[创建 UMG 控件模板（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/creating-umg-widget-templates-in-unreal-engine)
- 产出：`WBP_CustomButton` + `WBP_HUD` 复用它

### Day 5（显示/隐藏与生命周期：别把 UI 创建散落各处）
- 学：Widget 的创建、添加到 Viewport、移除、引用保存策略
- 做：实现一个“UI Toggle”（按键打开/关闭菜单），确保不会重复创建泄漏
- 链接：[UMG UI 设计器快速入门（回看显示/菜单部分）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/umg-ui-designer-quick-start-guide-in-unreal-engine)
- 产出：`ToggleMenu()`（可重复开关且稳定）

### Day 6（UI 架构：UMG Best Practices（Tech Blog））
- 学：如何把 UI 分层（数据/逻辑/表现）、避免蓝图堆成一团
- 做：读完后写 10 条“UI 规范”（事件驱动、减少绑定、数据模型独立）
- 链接：[UMG 最佳实践（Tech Blog）](https://www.unrealengine.com/zh-CN/tech-blog/umg-best-practices)
- 产出：`docs/ui-guidelines-week9.md`

### Day 7（官方 Best Practices（文档版）+ 小重构）
- 学：官方文档对 UMG 组织方式的建议
- 做：把本周 UI 做一次小重构：命名、目录、减少重复节点
- 链接：[UMG 最佳实践（官方文档）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/umg-best-practices-in-unreal-engine)
- 产出：一次 `refactor:` 提交（UI 目录/命名/复用）

#### 每周验收点
- 有 HUD + 主菜单；菜单可开关且不重复创建
- 至少 1 个数据项能稳定更新到 UI（绑定或事件驱动）
- 有一份 UI 规范文档（可用于面试描述）

#### 推荐练习题
- 做一个“Toast 提示”控件：接收事件 → 显示 2 秒 → 自动消失  
- 做一个“设置面板”：音量/难度变量更新后 UI 立即刷新

## Week 10｜数据驱动 UI + ListView/滚动列表 + MVVM（入门）
### 本周目标
- 用 DataTable/数据源驱动 UI（商店/背包原型）
- 形成“列表类 UI”的标准结构（Item/Entry/刷新策略）

### Day 1（数据驱动 UI 思路：DataTable + 商店示例）
- 学：Data-driven UI 的核心动机与结构
- 做：读完并建立 `DT_ShopItems`（至少 5 条）
- 链接：[使用 UMG 创建数据驱动型 UI（Tech Blog）](https://www.unrealengine.com/zh-CN/tech-blog/creating-a-data-driven-ui-with-umg)
- 产出：`DT_ShopItems` + `WBP_Shop`（能读表）

### Day 2（滚动列表：动态生成条目 + 选中事件）
- 学：动态生成条目、选中后广播事件、右侧详情面板更新
- 做：实现“左侧列表 + 右侧详情”交互（最小可用）
- 链接：[使用 UMG 创建数据驱动型 UI（继续实现列表与事件）](https://www.unrealengine.com/zh-CN/tech-blog/creating-a-data-driven-ui-with-umg)
- 产出：`WBP_ShopEntry` + `WBP_Shop`（选择条目更新详情）

### Day 3（UMG Viewmodel：了解 MVVM 插件与 View Binding）
- 学：UMG Viewmodel（MVVM）概念、View Binding、数组与 Views（ListView/TileView）
- 做：创建一个最小 Viewmodel（如 `VM_Shop`），绑定 1–2 个字段到 UI

### Day 4（对比：Property Binding vs View Binding）
- 学：属性绑定（函数/变量） vs MVVM 视图绑定（更结构化）
- 做：选一个字段（金币/库存），分别用两种方式实现，写对比结论
- 链接：
  - [UMG 属性绑定（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/property-binding-for-umg-in-unreal-engine)
  - [UMG Viewmodel（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/umg-viewmodel-for-unreal-engine)
- 产出：`docs/binding-compare-week10.md`

### Day 5（ListView 实战补充：Entry/Item 概念扫盲）
- 学：ListView 的 Item/Entry、刷新/复用的常见坑
- 做：把商店列表从“手动生成控件”迁移到 ListView（能跑通即可）
- 链接（补充读物，非官方但很实用）：
  - [UMG ListView 详解（知乎）](https://zhuanlan.zhihu.com/p/127184008)
- 产出：`WBP_Shop` 使用 ListView（最小跑通）

### Day 6（优化：事件驱动更新，减少频繁绑定/轮询）
- 学：UI 更新触发点：数据变更 → 事件/通知 → UI 刷新
- 做：把 1–2 个 UI 更新改为事件驱动（减少 Tick/频繁绑定）
- 链接：[UMG 最佳实践（Tech Blog）](https://www.unrealengine.com/zh-CN/tech-blog/umg-best-practices)
- 产出：一次 `perf/refactor:`（减少无意义刷新）

### Day 7（可选：3D UI（Widget Component））
- 学：WidgetComponent 在场景里做 3D UI
- 做：把一个按钮/面板放进 3D 空间（可交互不强求）
- 链接：[控件组件 Widget Component（官方）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/widget-components-in-unreal-engine)
- 产出：关卡里出现 3D UI（可选）

#### 每周验收点
- 商店/背包原型：**数据源驱动列表**，选择条目能更新详情
- 至少写出一份“绑定方式对比”结论（含你项目的取舍）
- ListView 最小跑通（哪怕功能简单）

#### 推荐练习题
- 背包 UI：支持排序/过滤（策略模式思路）  
- 商店 UI：购买后更新金币/库存（事件驱动，不靠 Tick）

## Week 11｜UI 系统化：分层、导航、管理器（为“插件化”做准备）
### 本周目标
- 形成一个“可维护 UI 系统”：UI 管理器/路由/栈式导航

- 做：写你项目的 3 层结构草图（Data/Logic/View）
- 链接：[UMG 最佳实践（Tech Blog）](https://www.unrealengine.com/zh-CN/tech-blog/umg-best-practices)
- 产出：`docs/ui-architecture-week11.md`

### Day 2（实现 UIManager：集中 Create/Show/Hide/Destroy）
- 学：把 UI 创建收口到一个“唯一入口”
- 做：实现 `UUIManager`（C++ 或 BP 均可），统一管理菜单/HUD
- 链接：[控件蓝图（官方，回看创建与组织方式）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/widget-blueprints-in-umg-for-unreal-engine)

### Day 5（可选：Editor Utility Widgets 做编辑器工具）
- 学：用 EUW 快速做编辑器面板（对系统/工具岗有加分）

### Day 6（整理规范：目录/命名/接口）
- 学：把系统变“可移植”：减少硬引用、统一入口、清晰依赖
- 做：整理 `UI/` 目录；写接口边界（哪些可 public，哪些必须 private）
- 链接：[UMG 最佳实践（官方文档）](https://dev.epicgames.com/documentation/zh-cn/unreal-engine/umg-best-practices-in-unreal-engine)
- 产出：`docs/ui-styleguide.md`

### Day 7（小型重构：为插件化做准备）
- 学：插件化前的预清理：收口依赖、抽接口
- 做：把 UIManager 的对外 API 收口（1–3 个入口函数即可）
- 链接：[UMG 最佳实践（Tech Blog）](https://www.unrealengine.com/zh-CN/tech-blog/umg-best-practices)
- 产出：一次 `refactor:`（为 Week13 插件化铺路）

#### 每周验收点
- 所有 UI 创建/切换都走 UIManager（不再散落各处）
- 页面栈可用（至少 2 层导航）
- 至少 1 个数据更新走事件驱动（非 Tick）

#### 推荐练习题
- 全局弹窗系统：Confirm/Cancel 回调（由 UIManager 管）  
- 红点系统：数据变化 → UI 更新（事件驱动）

## Week 13｜插件化：把 UI 系统做成 Plugin + 模块边界（Runtime/Editor）
